# 需求规格文档 - 医生认证隔离与功能修复

## 1. 概述

### 1.1 项目背景
当前系统存在以下问题：
1. 医生登入没有做隔离，登入状态可能过期
2. 没有形成独立的token机制
3. 家属页面与患者页面的聊天功能不正常
4. 添加功能出现错误，应该从数据库中选择而非手动输入

### 1.2 目标
- 实现医生登入的完整隔离机制
- 建立独立的token管理体系
- 修复家属和患者页面的聊天功能
- 改进添加功能，支持从数据库选择

## 2. 功能需求

### 2.1 医生登入隔离

#### 2.1.1 Token隔离机制
- **需求描述**：医生登入后应生成独立的token，与其他角色（家属、护士、管理员）完全隔离
- **验收标准**：
  - 医生token存储在独立的key中（doctorToken）
  - Token包含角色标识（subject: "doctor"）
  - Token过期时间独立管理
  - 不同角色的token互不干扰

#### 2.1.2 登录状态管理
- **需求描述**：医生登录状态应独立维护，包括：
  - 登录状态标识（isLoggedInAsDoctor）
  - 医生信息缓存（doctorInfo）
  - 登录过期检测机制
- **验收标准**：
  - 登录状态独立存储
  - 过期后自动跳转到医生登录页
  - 不影响其他角色的登录状态

#### 2.1.3 权限验证
- **需求描述**：医生访问相关页面时需验证医生权限
- **验收标准**：
  - 非医生用户无法访问医生专属页面
  - Token过期时自动跳转登录页
  - 权限验证失败有明确提示

### 2.2 聊天功能修复

#### 2.2.1 家属聊天功能
- **需求描述**：家属页面与医生的聊天功能应正常工作
- **验收标准**：
  - 家属能正常发送消息给医生
  - 能正常接收医生回复
  - 消息历史正确显示
  - 48小时过期机制正常工作
  - 消息轮询机制正常（3秒间隔）

#### 2.2.2 患者聊天功能
- **需求描述**：患者页面与医生的聊天功能应正常工作
- **验收标准**：
  - 患者能正常发送消息给医生
  - 能正常接收医生回复
  - 消息历史正确显示
  - 权限验证正确（仅患者可访问）

#### 2.2.3 消息隔离
- **需求描述**：不同角色的消息应完全隔离
- **验收标准**：
  - 家属只能看到自己与医生的消息
  - 患者只能看到自己与医生的消息
  - 消息不串角色

### 2.3 添加功能改进

#### 2.3.1 数据库选择支持
- **需求描述**：添加功能应支持从数据库中选择数据，而非仅手动输入
- **验收标准**：
  - 添加健康记录时可选择已有患者
  - 添加预约时可选择已有医生/医院
  - 选择列表从数据库动态加载
  - 支持搜索和筛选

#### 2.3.2 数据验证
- **需求描述**：添加的数据应进行有效性验证
- **验收标准**：
  - 必填字段验证
  - 数据格式验证
  - 数据库约束验证
  - 错误提示清晰

## 3. 非功能需求

### 3.1 安全性
- Token采用JWT标准，包含签名验证
- 敏感信息加密存储
- 防止token篡改和重放攻击

### 3.2 性能
- Token验证响应时间 < 100ms
- 消息加载时间 < 500ms
- 数据库查询优化

### 3.3 可维护性
- 代码结构清晰，职责分离
- 关键逻辑有注释说明
- 错误处理完善

## 4. 约束条件

### 4.1 技术约束
- 前端：HarmonyOS ArkTS
- 后端：Spring Boot + JWT
- 数据库：关系型数据库

### 4.2 兼容性约束
- 不影响现有其他角色的功能
- 保持API接口向后兼容
- 数据迁移平滑
