# 编码任务规划 - 医生认证隔离与功能修复

## 任务概览

本任务规划基于需求规格文档和实现方案文档，将医生认证隔离、聊天功能修复和添加功能改进分解为具体的编码任务。

## 阶段一：医生Token隔离实现

### 任务 1.1：后端JWT工具类增强
**优先级**：高  
**预估工作量**：1小时  
**文件位置**：`src/main/java/com/example/medical/common/JwtUtil.java`

**任务描述**：
- 为医生角色添加独立的JWT生成方法
- Token中包含`subject: "doctor"`标识
- 设置医生Token独立过期时间（7天）
- 添加Token角色验证方法

**验收标准**：
- [ ] 生成医生Token包含正确的subject标识
- [ ] Token过期时间设置为7天
- [ ] validateRole方法能正确验证角色

---

### 任务 1.2：创建前端Token管理器
**优先级**：高  
**预估工作量**：2小时  
**文件位置**：`entry/src/main/ets/common/utils/TokenManager.ets`（新建）

**任务描述**：
- 创建TokenManager统一管理类
- 实现getCurrentToken方法，按优先级获取Token（家属 > 护士 > 医生 > 普通用户）
- 实现saveDoctorToken方法，保存医生Token和相关信息
- 实现clearDoctorToken方法，清除医生登录状态
- 实现isDoctorLoggedIn方法，判断医生登录状态

**验收标准**：
- [ ] TokenManager能正确获取当前角色Token
- [ ] 医生Token独立存储在doctorToken key
- [ ] 登录状态正确维护

**预估工作量**：1.5小时  
**文件位置**：`entry/src/main/ets/common/utils/HttpUtil.ets`

**任务描述**：
- 修改getAuthorizationHeader方法，使用TokenManager获取Token
- 添加医生Token检测逻辑
- 修改401错误处理，根据角色跳转对应登录页

**验收标准**：

## 风险提示

1. **Token冲突风险**：修改Token管理逻辑时需充分测试，确保不影响现有角色
2. **消息隔离风险**：需严格测试消息隔离机制，防止消息串角色
3. **兼容性风险**：修改接口时需保持向后兼容，避免影响现有功能
4. **性能风险**：消息轮询和数据选择查询需注意性能优化
