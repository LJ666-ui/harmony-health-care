# 多端隔离医疗健康管理系统 - 任务规划文档

**版本**: v1.0
**创建日期**: 2025-01-15
**最后更新**: 2025-01-15
**作者**: System
**状态**: 草稿

## 任务规划概述

本文档将多端隔离医疗健康管理系统的技术设计转化为可执行的开发任务，按照优先级和依赖关系组织任务，确保开发过程有序进行。

**任务统计**：
- 主任务：12个
- 子任务：48个
- 覆盖需求：24个功能需求 + 15个非功能需求

**开发周期**：14周（Phase 1-5）

---

## 任务 1：项目基础架构搭建

**优先级**: P0
**预估工时**: 5天
**负责模块**: 全局
**依赖**: 无

### 任务描述

搭建HarmonyOS多端项目基础架构，创建五端独立应用入口，配置开发环境和构建流程。

### 子任务

#### 1.1 创建HarmonyOS项目结构

**输入**: 项目需求文档、技术选型方案

**输出**: 
- HarmonyOS项目目录结构
- 五端应用入口文件（DoctorEntry.ets、PatientEntry.ets、NurseEntry.ets、FamilyEntry.ets、AdminEntry.ets）
- 公共模块目录（common/）

**验收标准**:
- 项目结构符合设计文档定义
- 五端入口文件独立，互不依赖
- 公共模块可被各端引用

**实现提示**:
```
使用DevEco Studio创建HarmonyOS项目，配置module.json5定义五个Ability。
每个Ability对应一个终端应用，设置独立的icon和label。
创建common共享模块，存放公共组件、服务、工具类。
```

#### 1.2 配置构建和打包流程

**输入**: 项目结构

**输出**:
- build-profile.json5配置文件
- hvigorfile.ts构建脚本
- 多环境配置（dev、test、prod）

**验收标准**:
- 支持单独构建某一端应用
- 支持批量构建所有端
- 环境配置正确切换

**实现提示**:
```
配置hvigor构建脚本，定义task分别构建五个终端。
使用productFlavors配置不同环境。
配置签名证书用于发布。
```

#### 1.3 搭建后端服务基础框架

**输入**: 技术选型方案

**输出**:
- Node.js项目结构
- Express应用框架
- API Gateway基础代码

**验收标准**:
- 后端服务可启动
- API Gateway可路由请求
- 基础中间件配置完成（日志、错误处理）

**实现提示**:
```
使用Express创建API Gateway，配置路由、限流、日志中间件。
创建services目录，按业务领域划分微服务。
配置PM2用于进程管理。
```

#### 1.4 配置数据库连接

**输入**: 数据库设计文档

**输出**:
- MySQL数据库和表结构
- Redis连接配置
- InfluxDB初始化脚本
- MinIO存储桶配置

**验收标准**:
- 所有数据库连接成功
- 表结构创建完成
- 索引创建完成

**实现提示**:
```
编写SQL脚本创建数据库和表。
使用Sequelize或TypeORM作为ORM框架。
配置Redis连接池和InfluxDB客户端。
创建MinIO存储桶用于文件存储。
```

## 任务 2：用户认证与权限系统

**优先级**: P0
**预估工时**: 7天
**负责模块**: 认证模块
**依赖**: 任务1

### 任务描述

实现用户登录、Token管理、权限验证功能，支持JWT认证和生物识别登录。

### 子任务

#### 2.1 实现用户登录功能

**输入**: 用户凭证（用户名、密码、角色）

**输出**:
- 登录API接口（POST /api/auth/login）
- JWT Token生成和验证
- 登录页面UI（五端独立设计）

**验收标准**:
- 用户可使用用户名密码登录
- Token正确生成并包含用户信息和权限
- 登录失败返回明确错误信息
- 各端登录页面独立设计

**实现提示**:
```
后端：使用jsonwebtoken库生成Token，设置有效期2小时。
前端：使用ArkUI创建登录表单，调用登录API，存储Token到LocalStorage。
密码使用bcrypt加密存储和验证。
```

#### 2.2 实现Token刷新机制

**输入**: Refresh Token

**输出**:
- Token刷新API（POST /api/auth/refresh）
- Token过期自动刷新逻辑

**验收标准**:
- Token过期前自动刷新
- Refresh Token有效期7天
- Token刷新失败跳转登录页

**实现提示**:
```
在API拦截器中检查Token过期时间，提前5分钟自动刷新。
使用axios拦截器实现全局Token刷新逻辑。
```

#### 2.3 实现生物识别登录

**输入**: 用户生物特征（指纹/面容）

**输出**:
- 生物识别认证API（POST /api/auth/biometric）
- 生物识别登录UI

**验收标准**:
- 支持指纹和面容识别
- 生物特征绑定用户账号
- 生物识别失败降级到密码登录

**实现提示**:
```
使用HarmonyOS生物识别API（@ohos.userIAM_userAuth）。
首次登录后提示用户绑定生物特征。
存储生物特征Token到本地安全存储。
```

#### 2.4 实现RBAC权限控制

**输入**: 用户角色、资源、操作

**输出**:
- 权限定义配置
- 权限检查中间件
- 前端权限指令

**验收标准**:
- 不同角色有不同权限
- 无权限访问返回403错误
- 前端根据权限显示/隐藏功能

**实现提示**:
```
定义角色-权限映射表。
后端中间件检查用户权限。
前端使用自定义组件装饰器控制元素显示。
```

#### 2.5 实现多端登录隔离

**输入**: 用户角色、设备ID

**输出**:
- 角色与终端匹配验证
- 跨端登录拦截

**验收标准**:
- 医生只能在医生端登录
- 患者只能在患者端登录
- 错误终端登录提示错误

**实现提示**:
```
登录时验证角色与终端类型匹配。
Token中包含角色信息，前端验证Token角色与当前终端一致。
```

## 任务 3：跨端聊天功能

**优先级**: P0
**预估工时**: 8天
**负责模块**: 聊天模块
**依赖**: 任务2

### 任务描述

实现医生-患者、护士-患者、家属-医护的实时聊天功能，支持文字、图片、语音消息。

### 子任务

#### 3.1 搭建WebSocket通信层

**输入**: WebSocket协议设计

**输出**:
- WebSocket服务端（Socket.io）
- WebSocket客户端（前端服务）
- 连接管理和心跳机制

**验收标准**:
- WebSocket连接稳定
- 支持断线重连
- 心跳检测正常工作

**实现提示**:
```
后端使用Socket.io创建WebSocket服务。
前端创建WebSocketService封装连接、断线重连逻辑。
设置30秒心跳间隔，60秒超时。
```

#### 3.2 实现消息发送和接收

**输入**: 聊天消息（文字、图片、语音）

**输出**:
- 发送消息API（WebSocket事件）
- 接收消息回调
- 消息状态管理（发送中、已发送、已读）

**验收标准**:
- 消息实时送达
- 显示发送状态
- 消息按时间排序

**实现提示**:
```
定义SEND_MESSAGE和MESSAGE_RECEIVED事件。
消息发送后先显示本地，WebSocket确认后更新状态。
使用消息队列保证消息顺序。
```

#### 3.3 实现聊天会话管理

**输入**: 用户ID、联系人列表

**输出**:
- 会话列表API（GET /api/chat/sessions）
- 会话列表UI
- 未读消息计数

**验收标准**:
- 显示所有聊天会话
- 按最后消息时间排序
- 显示未读消息数量

**实现提示**:
```
后端查询用户参与的所有会话，按更新时间排序。
前端使用List组件渲染会话列表。
Redis存储未读消息计数。
```

#### 3.4 实现聊天历史查询

**输入**: 会话ID、分页参数

**输出**:
- 历史消息API（GET /api/chat/messages/:sessionId）
- 历史消息UI
- 上拉加载更多

**验收标准**:
- 显示完整历史消息
- 支持分页加载
- 支持按日期搜索

**实现提示**:
```
后端查询消息表，按时间倒序分页返回。
前端使用虚拟滚动优化长列表性能。
实现上拉加载更多逻辑。
```

#### 3.5 实现图片和语音消息

**输入**: 图片文件、语音文件

**输出**:
- 文件上传API（POST /api/chat/upload）
- 图片预览组件
- 语音录制和播放组件

**验收标准**:
- 支持图片发送和预览
- 支持语音录制和播放
- 文件大小限制（图片5MB、语音1MB）

**实现提示**:
```
使用MinIO存储文件，生成临时访问URL。
图片使用Image组件，支持点击放大。
语音使用AudioRecorder和AudioPlayer API。
```

#### 3.6 实现消息已读状态

**输入**: 会话ID、消息ID

**输出**:
- 标记已读API（WebSocket事件）
- 已读状态UI（双勾标记）

**验收标准**:
- 消息已读显示双勾
- 对方已读实时更新状态
- 进入聊天页面自动标记已读

**实现提示**:
```
定义MARK_READ事件，发送已读确认。
后端更新消息已读状态，通知发送方。
前端根据已读状态显示不同图标。
```

## 任务 4：医疗服务核心功能

**优先级**: P0
**预估工时**: 10天
**负责模块**: 医疗服务模块
**依赖**: 任务2

### 任务描述

实现病历查看、处方开具、诊断记录等核心医疗功能，支持医嘱自动联动。

### 子任务

#### 4.1 实现病历查看功能

**输入**: 患者ID

**输出**:
- 病历查询API（GET /api/medical/records/:patientId）
- 病历详情页面UI
- 病历数据展示组件

**验收标准**:
- 显示患者完整病历
- 包含基本信息、既往病史、过敏史、检查报告、处方记录
- 医生和患者可查看（权限不同）

**实现提示**:
```
后端查询medical_records表及相关表，组装完整病历数据。
前端使用Tab组件分类展示不同信息。
患者查看自己的病历时部分敏感信息脱敏。
```

#### 4.2 实现处方开具功能

**输入**: 患者ID、药品信息、用法用量

**输出**:
- 开处方API（POST /api/medical/prescriptions）
- 处方表单UI
- 处方合理性验证

**验收标准**:
- 医生可开具处方
- 验证药品相互作用
- 处方保存成功触发用药提醒

**实现提示**:
```
创建处方表单，包含药品选择、用法用量输入。
调用药品相互作用验证API（可接入第三方）。
保存处方后发送消息队列事件，触发提醒生成。
```

#### 4.3 实现用药提醒自动生成

**输入**: 处方信息

**输出**:
- 提醒生成逻辑
- 提醒数据（medication_reminders表）
- 提醒推送通知

**验收标准**:
- 根据处方自动生成提醒
- 提醒时间计算正确
- 患者端收到提醒通知

**实现提示**:
```
监听处方创建事件，调用calculateReminderTimes算法。
批量插入提醒记录到数据库。
推送通知到患者端，显示在提醒列表。
```

#### 4.4 实现用药提醒定时推送

**输入**: 提醒时间

**输出**:
- 定时任务调度器
- 提醒推送逻辑
- 提醒通知UI

**验收标准**:
- 到达设定时间推送提醒
- 提醒包含药品信息
- 支持确认、延迟、跳过操作

**实现提示**:
```
使用node-cron或Agenda创建定时任务。
每分钟扫描即将到期的提醒，推送通知。
前端接收通知，显示提醒弹窗。
```

#### 4.5 实现服药确认功能

**输入**: 提醒ID、操作类型（确认/延迟/跳过）

**输出**:
- 确认服药API（POST /api/medical/reminders/:id/confirm）
- 服药记录更新
- 医护端同步显示

**验收标准**:
- 患者可确认服药
- 记录服药时间和状态
- 医生和护士可查看服药记录

**实现提示**:
```
更新提醒状态和实际服药时间。
发送消息队列事件，同步到医护端。
医护端显示患者用药记录列表。
```

#### 4.6 实现诊断记录功能

**输入**: 患者ID、症状、检查结果、诊断结论

**输出**:
- 诊断记录API（POST /api/medical/diagnoses）
- 诊断表单UI
- 诊断记录展示

**验收标准**:
- 医生可记录诊断
- 诊断保存到病历
- 包含症状、检查结果、诊断结论、治疗方案

**实现提示**:
```
创建诊断表单，支持症状多选、检查结果输入。
保存诊断记录，关联患者病历。
前端使用卡片组件展示诊断记录。
```

## 任务 5：紧急通知系统

**优先级**: P0
**预估工时**: 6天
**负责模块**: 通知模块
**依赖**: 任务2、任务3

### 任务描述

实现突发情况自动检测和紧急通知多端推送，保证通知送达。

### 子任务

#### 5.1 实现异常健康数据检测

**输入**: 患者健康数据（心率、血压、体温等）

**输出**:
- 数据阈值配置
- 异常检测逻辑
- 异常事件触发

**验收标准**:
- 实时检测健康数据
- 超出阈值触发异常事件
- 标记异常类型和严重程度

**实现提示**:
```
配置健康指标正常范围（心率60-100、血压90-140等）。
接收健康设备数据，实时检测是否异常。
异常时发送消息队列事件，触发紧急通知流程。
```

#### 5.2 实现紧急通知多端推送

**输入**: 紧急事件信息

**输出**:
- 紧急通知API（POST /api/notifications/emergency）
- 多端推送逻辑
- 通知送达保证

**验收标准**:
- 护士、医生、家属同时收到通知
- 通知包含患者位置、异常指标、病历摘要
- 推送失败自动重试3次

**实现提示**:
```
查询患者关联的护士、医生、家属。
并行推送WebSocket消息到各端。
失败时使用指数退避重试，最多3次。
记录推送日志和状态。
```

#### 5.3 实现患者紧急求助功能

**输入**: 患者求助请求

**输出**:
- 求助按钮UI
- 求助API
- 求助通知推送

**验收标准**:
- 患者可一键求助
- 立即通知护士、医生、家属
- 显示患者位置和状态

**实现提示**:
```
在患者端首页显著位置放置求助按钮。
点击后弹出确认对话框，确认后发送求助请求。
后端触发紧急通知流程，推送通知。
```

#### 5.4 实现紧急通知UI

**输入**: 紧急通知数据

**输出**:
- 紧急告警组件（EmergencyAlertComponent）
- 告警声音提示
- 告警处理界面

**验收标准**:
- 紧急通知醒目显示（红色背景、闪烁）
- 自动播放告警声音
- 支持查看详情和处理

**实现提示**:
```
使用全屏弹窗显示紧急通知。
播放系统告警音或自定义音频。
提供"查看详情"和"立即处理"按钮。
```

#### 5.5 实现通知列表和状态管理

**输入**: 用户ID

**输出**:
- 通知列表API（GET /api/notifications）
- 通知列表UI
- 已读/未读状态

**验收标准**:
- 显示所有通知
- 区分普通通知和紧急通知
- 支持标记已读

**实现提示**:
```
查询notifications表，按时间倒序返回。
前端使用List组件，紧急通知特殊样式。
点击通知标记已读，更新状态。
```

## 任务 6：家属功能模块

**优先级**: P0
**预估工时**: 5天
**负责模块**: 家属端
**依赖**: 任务2、任务4

### 任务描述

实现家属查看患者详情、提交健康信息、授权管理功能。

### 子任务

#### 6.1 实现家属查看患者详情

**输入**: 患者ID、家属授权等级

**输出**:
- 患者详情API（GET /api/patients/:id/detail）
- 患者详情页面UI
- 数据权限过滤

**验收标准**:
- 家属可查看授权患者信息
- 根据授权等级显示不同信息
- 显示健康指标、就医记录、用药情况

**实现提示**:
```
验证家属授权等级，过滤敏感信息。
查询患者健康数据、病历、用药记录。
前端使用卡片和图表展示数据。
```

#### 6.2 实现家属健康信息提交

**输入**: 健康信息（症状、检查报告、家族病史）

**输出**:
- 提交API（POST /api/medical/family-health）
- 提交表单UI
- 附件上传

**验收标准**:
- 家属可提交健康信息
- 支持附件上传
- 医生端收到并关联患者病历

**实现提示**:
```
创建健康信息提交表单，支持多种类型。
上传附件到MinIO，保存URL。
发送消息队列事件，通知医生端。
医生端显示家属健康信息列表。
```

#### 6.3 实现家属授权管理

**输入**: 患者ID、家属ID、授权等级

**输出**:
- 授权设置API（POST /api/patients/:id/authorization）
- 授权管理UI
- 授权验证逻辑

**验收标准**:
- 患者可设置家属查看权限
- 支持分级授权（1-3级）
- 家属仅能查看授权范围信息

**实现提示**:
