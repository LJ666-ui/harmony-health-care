📌 GitHub 分支保护规则 一步一步操作指南
（完全对应你当前的仓库，跟着点就行，零出错）
一、先回到仓库首页
点击页面左上角的仓库名（比如 harmony-health-care），回到仓库主页。
确认你在 Code 标签页（默认就是）。
二、进入分支保护设置页面
点击页面上方的 Settings（⚙️ 设置按钮，在 Code/Issues/Pull requests 最右边）。
左侧菜单栏往下滑，找到 Code and automation 分组，点击 Branches。
进入后，页面中间会显示 Branch protection rules 区域，点击 Add rule（添加规则）按钮。
三、配置 main 分支保护规则（第一步，先保护主分支）
1. 填写分支名称
在 Branch name pattern 输入框里，直接输入：main（系统会自动匹配你仓库的 main 分支，输入后会显示 ✅ 匹配成功）
2. 开启核心保护规则（重点！）
往下滑，在 Protect matching branches 区域，必须勾选以下2项：
✅ Require a pull request before merging（合并前必须有Pull Request）
勾选后，会展开子选项，必须勾选：
✅ Require approvals（需要审核批准）
把 Required number of approvals before merging 改成 1（1人审核即可，队长审核）
其他子选项（如 dismiss stale pull request approvals）保持默认不勾选
✅ Do not allow bypassing the above settings（禁止绕过上述设置，超级重要！）
这个选项会强制所有人都遵守规则，包括你自己，防止误操作
3. 其他可选规则（推荐开启，更安全）
可以额外勾选：
✅ Require status checks to pass before merging（合并前检查通过，可选，初期可以不开）
✅ Require conversation resolution before merging（合并前解决所有评论，可选）
4. 保存规则
拉到页面最底部，点击 Create（创建）按钮，main 分支保护就完成了！
四、配置 dev 分支保护规则（第二步，保护开发主干）
回到 Branches 设置页，再次点击 Add rule。
Branch name pattern 输入框里，输入：dev（系统会自动匹配你仓库的 dev 分支，显示 ✅ 匹配成功）
完全重复上面 main 分支的勾选配置：
✅ Require a pull request before merging
✅ Require approvals（1人审核）
✅ Do not allow bypassing the above settings
拉到页面最底部，点击 Create 按钮，dev 分支保护完成！

三、最终推荐：方案 2（分大文件夹），强烈建议现在就改
为什么现在改最好？
现在是项目初始化阶段，改结构成本为 0：只有前端初始化代码，没有复杂业务，一次迁移就搞定，后期代码多了再改会非常麻烦
一劳永逸解决团队协作问题：前后端开发完全隔离，前端改前端，后端改后端，再也不会出现文件冲突、代码混乱
符合行业标准规范：所有正规前后端分离项目，都是这种结构，竞赛评审时会直接加分，显得专业
不影响现有代码运行：只是把文件挪个文件夹，DevEco、IDEA 都能正常识别，不会出任何问题
四、如果坚持用方案 1（散列结构），会有什么问题？
后端代码只能散在根目录，和前端文件混在一起，找文件、排查问题、代码合并都会非常痛苦
团队协作时，前后端开发容易互相改到对方的文件，产生大量冲突
竞赛评审时，根目录一堆零散文件，显得项目不规范、不专业
后期项目迭代，结构会越来越乱，维护成本指数级上升
