# 星云医疗助手 - 项目设计方案

## 一、总体架构设计

### 1.1 架构概览

本项目采用 **前后端分离 + 分层架构** 的设计方案，前端基于 HarmonyOS ArkTS 构建，后端基于 Spring Boot 提供 RESTful API 服务。整体架构分为以下层次：

```
┌─────────────────────────────────────────────────────────────┐
│                      表现层 (Presentation)                    │
│  Pages (100+) → Components (50+) → Charts → Accessible UI   │
├─────────────────────────────────────────────────────────────┤
│                      业务逻辑层 (Business)                    │
│  AI Agent │ SmartWard │ AR Nav │ DigitalTwin │ Distributed   │
├─────────────────────────────────────────────────────────────┤
│                      数据管理层 (Data)                        │
│  Store │ Manager │ ViewModel │ RDB │ Preferences │ Sync      │
├─────────────────────────────────────────────────────────────┤
│                      基础设施层 (Infrastructure)              │
│  HttpUtil │ RouterUtil │ Accessibility │ Performance │ Mock  │
├─────────────────────────────────────────────────────────────┤
│                      后端服务层 (Backend)                     │
│  Spring Boot REST API (100+ 接口)                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 每个模块/类只负责一项功能，如 DeviceManager 只管设备、AlertManager 只管告警 |
| 单例模式 | 核心服务类采用单例模式（DistributedManager、ParkingService、RdbHelper 等） |
| 接口隔离 | 通过接口定义契约（IDeviceController、IRuleExecutor），实现解耦 |
| 分层解耦 | 表现层/业务层/数据层/基础设施层各司其职，层间通过接口通信 |
| 响应式设计 | 基于 BreakpointSystem 实现多断点响应式布局 |
| 无障碍优先 | 全局无障碍配置，老年模式/字体缩放/高对比度一体化设计 |
| 离线优先 | 本地 RDB 缓存 + 同步队列，保障弱网/离线场景可用性 |

## 二、模块架构设计

### 2.1 AI 多智能体系统架构

```
用户输入
   │
   ▼
┌──────────────────┐
│ IntentClassifier │ ← 意图分类（内科/眼科/全科等）
└────────┬─────────┘
         │
         ▼
┌──────────────────────┐
│ MultiAgentOrchestrator│ ← 多智能体编排器
│  ┌─────────────────┐ │
│  │ ConversationMemory│ ← 对话记忆（多轮上下文）
│  └─────────────────┘ │
│  ┌─────────────────┐ │
│  │  RAG Retriever   │ ← 知识检索增强
│  │  ┌─────────────┐│ │
│  │  │HybridRetriever││ ← 混合检索
│  │  │┌───────────┐││ │
│  │  ││VectorRetr. │││ ← 向量检索
│  │  │└───────────┘││ │
│  │  │┌───────────┐││ │
│  │  ││KeywordRetr.│││ ← 关键词检索
│  │  │└───────────┘││ │
│  │  │┌───────────┐││ │
│  │  ││ Reranker   │││ ← 重排序
│  │  │└───────────┘││ │
│  │  └─────────────┘│ │
│  └─────────────────┘ │
│  ┌─────────────────┐ │
│  │ Specialist Agents │ ← 专科智能体
│  │ ┌───────────────┐│ │
│  │ │NeurologyAgent ││ │ ← 神经内科
│  │ └───────────────┘│ │
│  │ ┌───────────────┐│ │
│  │ │OphthalmologyAg││ │ ← 眼科
│  │ └───────────────┘│ │
│  └─────────────────┘ │
└──────────┬───────────┘
           │
           ▼
      会诊结果聚合输出
```

**关键设计决策**：
- **意图分类先行**：用户输入先经过 IntentClassifier 分类，再路由到对应专科智能体
- **RAG 增强检索**：采用混合检索策略（向量 + 关键词），经 Reranker 重排序后注入上下文
- **对话记忆管理**：ConversationMemory 维护多轮对话上下文，支持会话级记忆
- **结果聚合**：MultiAgentOrchestrator 汇总各智能体回答，生成综合会诊报告

## 三、数据流设计

### 3.1 前后端数据交互

```
┌──────────┐     HTTP Request      ┌──────────────┐
│  ArkTS   │ ──────────────────→   │  Spring Boot  │
│  前端     │ ←──────────────────  │  后端 API     │
│          │     HTTP Response     │              │
└──────────┘                       └──────────────┘
     │                                    │
     ▼                                    ▼
┌──────────┐                       ┌──────────────┐
│   RDB    │                       │   MySQL DB   │
│ 本地缓存  │                       │   数据库      │
└──────────┘                       └──────────────┘
     │
     ▼
┌──────────┐
│SyncQueue │ ← 离线操作入队，网络恢复后自动同步
│ 同步队列  │
└──────────┘
```

**API 交互规范**：
- 统一 BaseResponse 封装（code / message / data）
- Token 认证（请求头携带 Authorization）
- 环境配置切换（DEV / TEST / PROD）
- 网络异常自动降级为本地 Mock 数据

### 3.2 离线数据同步策略

```
在线操作 → 直接请求后端 API
                │
                ├─ 成功 → 更新本地缓存
                │
                └─ 失败 → 写入 SyncQueue
                              │
                              ▼
                     网络恢复后自动重试
                              │
                              ▼
                     同步成功 → 清除队列记录
```

### 3.3 本地数据存储设计

| 存储方式 | 用途 | 数据示例 |
|---------|------|---------|
| RDB (SQLite) | 结构化数据缓存 | 用户信息、健康记录、文章缓存、设备缓存 |
| Preferences | 键值对配置 | 用户设置、登录状态、老年模式开关 |
| AppStorage | 运行时全局状态 | 当前断点、字体等级、老年模式状态 |
| SyncQueue | 离线操作队列 | 待同步的 CRUD 操作 |

## 五、性能优化方案

### 5.1 启动优化

```
App 启动
   │
   ▼
EntryAbility.onCreate()
   ├── initAccessibilitySettings()  ← 并行初始化
   ├── initDatabase()               ← 并行初始化
   └── initSettingsUtil()           ← 并行初始化
   │
   ▼
StartupOptimizer
   ├── 延迟加载非关键模块
   └── CriticalDataPreloader 预加载关键数据
```

### 5.2 渲染优化

| 优化项 | 方案 |
|-------|------|
| 组件复用 | ComponentPool 组件池，避免重复创建销毁 |
| 懒加载 | LazyDataSource + ComponentLazyLoader |
| 页面缓存 | PageCache 缓存已渲染页面 |
| 渲染优化 | RenderOptimizer 减少不必要的重绘 |
| 状态更新 | StateUpdateQueue 批量合并状态更新 |

### 5.3 内存优化

| 优化项 | 方案 |
|-------|------|
| 对象追踪 | ObjectTracker 追踪对象生命周期 |
| 图片缓存 | ImageCache LRU 缓存策略 |
| 内存回收 | MemoryOptimizer 主动触发 GC |
| 数据源优化 | LazyDataSource 按需加载数据 |

### 5.4 网络优化

| 优化项 | 方案 |
|-------|------|
| 请求合并 | RequestBatch 批量请求合并 |
| 离线缓存 | OfflineCache 离线数据缓存 |
| 缓存预热 | CacheWarmupLoader 预热缓存 |
| 网络优化 | NetworkOptimizer 请求调度优化 |
