# 🔗 星云医疗助手 - 前后端接口对接时间表（通俗版 V5.0）

> **版本说明**：基于前后端独立规划整合，聚焦"什么时候开始连接"
>
> **适用角色**：全体团队（前端B + 后端C + 队长A）
>
> **核心问题**：每天哪些接口需要对接？谁先写完？什么时候联调？

---

## 📌 一、这份文档是干什么的？

**简单说：这是前后端的"接头暗号表"**

之前已经有了：
- 后端C的"后端规划"（他写什么接口）
- 前端B的"前端规划"（她做什么页面）
- 联合规划（怎么配合）

**但还缺一个关键信息**：
- **具体哪个接口在什么时候对接？**
- **C写完后B什么时候开始用？**
- **每个接口的格式是什么？**
- **对接时遇到问题找谁？**

**这份文档就是回答这些问题的！**

它像一张"时刻表"，告诉你：
- 每天早上C要交出什么接口
- B拿到接口后立刻做什么
- 对接不上怎么办
- 验收标准是什么

#### 2️⃣ **components/** - 可复用组件库（11个组件）

**这是什么？**
像乐高积木一样的UI组件，可以在多个页面重复使用。避免每个页面都重复写相同的代码。

**有什么用？**
- 提高开发效率（写一次，到处用）
- 保持界面一致性（所有页面的按钮长得一样）
- 方便维护（改一处，全局生效）

**实现了什么功能？**

| 组件名 | 功能描述 | 使用场景 | 示例 |
|--------|---------|---------|------|
| [Header.ets](entry/src/main/ets/components/Header.ets) | 顶部导航栏 | 几乎所有页面 | 显示标题+返回按钮 |
| [Footer.ets](entry/src/main/ets/components/Footer.ets) | 底部导航栏 | 主页面 | 首页/医疗/康复/我的 4个Tab |
| [Button.ets](entry/src/main/ets/components/Button.ets) | 统一按钮 | 全局通用 | 登录按钮/提交按钮/确认按钮 |
| [Input.ets](entry/src/main/ets/components/Input.ets) | 统一输入框 | 表单页面 | 手机号输入/密码输入/搜索框 |
| [Card.ets](entry/src/main/ets/components/Card.ets) | 卡片容器 | 列表展示 | 健康记录卡片/医院卡片/文章卡片 |
| [ClickableCard.ets](entry/src/main/ets/components/ClickableCard.ets) | 可点击卡片 | 列表项 | 点击跳转详情的卡片 |
| [EmptyState.ets](entry/src/main/ets/components/EmptyState.ets) | 空状态提示 | 无数据时 | "暂无记录"/"加载失败" |
| [Loading.ets](entry/src/main/ets/components/Loading.ets) | 加载动画 | 数据加载中 | 转圈圈等待效果 |
| [PullRefresh.ets](entry/src/main/ets/components/PullRefresh.ets) | 下拉刷新 | 列表页面 | 下拉更新最新数据 |

**适老化特性**：
- 所有组件支持 `isElderMode` 参数

### 🔧 后端详细结构（Spring Boot服务）

#### 1️⃣ **controller/** - 控制器层（8个控制器）

**这是什么？**
后端的"前台接待员"，负责接收前端发来的HTTP请求，然后调用相应的业务逻辑。

**有什么用？**
- 定义API接口路径（如 `/user/login`）
- 接收请求参数（从URL或Body中提取）
- 返回响应结果（JSON格式）
- 处理跨域请求（CORS配置）

**实现了什么功能？**

| 控制器 | 负责模块 | API数量 | 核心接口 | 优先级 |
|--------|---------|---------|---------|--------|
| [UserController.java](src/main/java/com/example/medical/controller/UserController.java) | 用户认证 | 5个 | 注册/登录/获取信息/修改密码 | 🔴P0 |
| [HealthRecordController.java](src/main/java/com/example/medical/controller/HealthRecordController.java) | 健康记录 | 7个 | CRUD+批量导入+统计查询 | 🔴P0 |
| [HospitalController.java](src/main/java/com/example/medical/controller/HospitalController.java) | 医院服务 | 6个 | 搜索/筛选/距离计算 | 🟡P1 |
| [RehabPlanController.java](src/main/java/com/example/medical/controller/RehabPlanController.java) | 康复计划 | 5个 | 计划CRUD+进度跟踪 | 🟡P1 |
| [HealthArticleController.java](src/main/java/com/example/medical/controller/HealthArticleController.java) | 科普文章 | 5个 | 文章浏览/收藏/分类查询 | 🟡P1 |
| [HealthFoodController.java](src/main/java/com/example/medical/controller/HealthFoodController.java) | 饮食食谱 | 4个 | 食谱推荐/营养分析 | 🟢P2 |
| [MedicineController.java](src/main/java/com/example/medical/controller/MedicineController.java) | 药品管理 | 4个 | 药品信息/用药提醒 | 🟢P2 |

**工作原理示例**：
```
前端发送: POST http://localhost:8080/user/login
    ↓
UserController.login() 接收请求
    ↓
调用 UserService 进行密码验证
    ↓
返回: {"code":200,"message":"登录成功","data":{"token":"xxx","user":{...}}}
```

#### 2️⃣ **service/** - 业务逻辑层（8个服务接口+8个实现类）

**这是什么？**
后端的"大脑"，包含所有的业务规则和算法实现。

**有什么用？**
- 实现复杂的业务逻辑（如风险评估算法）

#### 3️⃣ **mapper/** - 数据访问层（8个Mapper接口）

**这是什么？**
后端的"数据库管家"，负责执行SQL语句与MySQL交互。

**有什么用？**
- 将Java对象映射到数据库表（ORM）
- 提供CRUD基本操作（MyBatis-Plus自动生成）
- 编写复杂SQL查询（多表关联/统计）
- 性能优化（索引利用/分页查询）

**实现了什么功能？**

| Mapper | 对应表 | 主要操作 | SQL复杂度 |
|--------|--------|---------|-----------|
| UserMapper | user表 | 用户CRUD | ⭐ 简单 |
| HealthRecordMapper | health_record表 | 记录CRUD+统计 | ⭐⭐ 中等 |
| HospitalMapper | hospital表 | 地理搜索 | ⭐⭐ 中等 |
| RehabPlanMapper | rehab_plan表 | 计划管理 | ⭐ 简单 |
| HealthArticleMapper | article表 | 文章查询+收藏 | ⭐⭐ 中等 |
| HealthFoodMapper | health_food表 | 食谱查询 | ⭐ 简单 |
| MedicineMapper | medicine表 | 药品查询 | ⭐ 简单 |

**技术栈**：MyBatis-Plus（简化SQL编写，提供开箱即用的CRUD方法）

---

#### 4️⃣ **entity/** - 实体类（7个实体）

**这是什么？**
数据库表的Java对象映射，每张表对应一个实体类。

**有什么用？**
- 字段映射（表列→Java属性）
- 类型转换（VARCHAR→String, INT→Integer）
- 数据校验（注解验证）
- JSON序列化（返回给前端时自动转换）

**实体列表**：

| 实体类 | 对应数据库表 | 关键字段 | 说明 |
|--------|-------------|---------|------|
| User | user | id/username/password/userType | 用户基本信息 |

**Result.java 的核心作用**：

```java
// 统一响应格式
public class Result<T> {
    private int code;        // 200=成功 400=参数错误 401=未授权 500=服务器错误
    private String message;  // 提示信息："登录成功"/"用户名已存在"
    private T data;          // 实际数据：用户对象/列表/null
}

// 使用示例
return Result.success("登录成功", user);        // {"code":200,"message":"登录成功","data":{...}}
return Result.error("用户名已存在");              // {"code":500,"message":"用户名已存在","data":null}
```

**为什么重要？**
- 前端只需要判断 `code == 200` 就知道是否成功
- 统一错误格式，前端可以统一处理错误弹窗
- 包含泛型 `<T>`，可以返回任意类型的数据

async handleLogin() {
  // 1. 表单校验
  if (!this.username || !this.password) {
    showToast('请输入手机号和密码')
    return
  }

  // 2. 显示加载状态
  this.isLoading = true

  try {
    // 3. 调用后端登录接口
    const response = await HttpUtil.post('/user/login', {
      username: this.username,
      password: this.password
    })

    // 4. 登录成功
    if (response.code === 200) {
      // 5. 保存Token
      AuthManager.saveToken(response.data.token)

      // 6. 保存用户基本信息
      AppStorage.setOrCreate('userInfo', response.data)

      // 7. 跳转首页
      router.pushUrl({ url: 'pages/HomePage' })
    } else {
      // 8. 登录失败显示错误
      showToast(response.message || '登录失败')
    }
  } catch (error) {
    // 9. 网络异常
    showToast('网络异常，请稍后重试')
  } finally {
    this.isLoading = false
  }
}

#### 接口4：Token管理机制

**这不是一个单独的接口，而是贯穿所有请求的机制**

**C的实现**：
- JwtInterceptor拦截器：每次请求检查Header中的Authorization
- Token有效 → 放行
- Token无效或缺失 → 返回401

**B的实现**：
- AuthManager.saveToken(token) → 存入本地存储
- HttpUtil每次请求自动读取Token附加到Header
- 收到401响应 → 清除Token → 跳转登录页

async sendMessage() {
  const question = this.inputText.trim()
  if (!question) return

  // 1. 立即显示用户消息（乐观更新）
  this.messages.push({
    role: 'user',
    content: question,
    time: new Date().toLocaleTimeString()
  })
  this.inputText = ''
  this.scrollToBottom()

  // 2. 显示AI思考动画
  this.messages.push({
    role: 'assistant',
    content: '',
    loading: true,
    time: ''
  })

      history: this.getRecentHistory(10),  // 最近10条消息
      context: this.userHealthContext      // 用户健康档案
    })

    // 4. 移除loading消息，显示真实回复
    this.messages.pop()
    this.messages.push({
      role: 'assistant',
      content: res.data.reply,
      time: new Date().toLocaleTimeString(),
      isMarkdown: true
    })

    // 5. 渲染Markdown
    this.renderMarkdown(res.data.reply)

  } catch (error) {
    // 6. 错误处理
    this.messages.pop()
    this.messages.push({
      role: 'assistant',
      content: '抱歉，发生了错误，请稍后重试',
      isError: true
    })

## ⚠️ 五、对接常见问题与解决方案

### 问题1：字段名不一致（概率70%）⚠️ 最高发

**现象**：C返回 `userId`，B期望 `user_id` 或 `id`

**解决**：
- Day 1对齐会就约定命名规范（驼峰camelCase）
- 共享文档实时更新字段定义
- 发现不一致立即微信沟通，不要自己猜

### 问题2：数据类型不匹配（概率50%）

**现象**：C返回数字 `age: 65`，B期望字符串 `"65"`

**解决**：
- 接口文档标注每个字段的类型（string/number/boolean/array/object）
- B做好类型转换（Number() / String()）
- 日期统一用ISO格式或时间戳

### 问题3：空值处理不当（概率60%）

**现象**：C返回 `null`，B没判断直接渲染导致崩溃

**解决**：
- B对所有可选字段做空值判断（`data?.field ?? defaultValue`）
- C对必填字段做校验（@NotNull注解）
- 空数组返回 `[]` 而不是 `null`

### 问题4：接口响应慢（概率40%）

**现象**：首页加载超过3秒

**解决**：
- C检查SQL慢查询（EXPLAIN分析）
- C添加数据库索引
- B加上骨架屏/Loding动画提升感知速度
- 非首屏数据延迟加载

### 问题5：跨域问题（概率30%）

**现象**：B调用C的接口报 CORS 错误

**解决**：
- C配置CorsConfig允许前端域名
- 开发环境允许所有来源（仅开发！）
- 生产环境指定白名单

### 问题6：Token过期时机（概率35%）

**现象**：用户操作过程中突然被踢出登录

**解决**：
- Token有效期设为24小时（不要太短）
- B收到401后先尝试刷新Token（如果有refresh_token）
- 跳转登录页前保存当前路径，登录后回到原页面
- 友好提示："登录已过期，请重新登录"
