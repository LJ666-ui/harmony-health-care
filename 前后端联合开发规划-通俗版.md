### ⚠️ 协作常见问题及解决方案

| 问题场景 | 原因 | 解决方案 | 责任人 |
|----------|------|---------|--------|
| **B调用C的接口报404** | C还没部署或路径写错 | C检查Controller的@RequestMapping注解 | C |
| **B收到数据但字段名不对** | C返回camelCase，B期望snake_case | 统一使用camelCase（Java标准） | A协调 |
| **C改了接口没通知B** | 缺乏沟通机制 | 修改接口必须在群里通知+更新接口文档 | C |
| **B前端页面空白** | C的服务没启动或跨域失败 | C启动Spring Boot + 检查CorsConfig | C |
| **Token验证总是失败** | B发送Token的格式不对 | 统一使用 `Bearer xxx` 格式 | B+C |
| **联调时发现缺字段** | 需求理解不一致 | 参考sql/medical_health.sql确认字段 | A协调 |

## 📆 第1周：基础设施搭建 + 核心流程打通

**本周目标**：用户能登录注册 → 看首页 → 记录健康数据 → 查看风险评估结果

**关键依赖关系**：
- ❗ **Day 1**：后端C必须先启动数据库和基础项目，前端B才能开始做HTTP请求
- ❗ **Day 3**：前后端必须对齐第一批接口格式（登录、首页、健康记录），否则后面白做
- ✅ 其他时间：大部分可以**并行开发**，互不干扰

### 🗓️ Day 1：项目启动 + 基础设施搭建（最重要的一天！）

#### 🌅 上午（9:00-12:00）：全员技术对齐会（2小时）

**参会人**：A（队长）+ B（前端）+ C（后端）

**会议内容**（必须完成！）：

1. **确认技术栈统一**（30分钟）
   - 后端确认：Spring Boot版本、MySQL端口、MyBatis-Plus配置
   - 前端确认：HarmonyOS SDK版本、DevEco Studio版本
   - 对齐：HTTP请求方式（RESTful API）、数据格式（JSON）、编码（UTF-8）

2. **确定接口规范**（45分钟）
   - 统一响应格式：code/message/data结构
   - 错误码约定：200=成功、400=参数错、401=未登录、403=无权限、500=服务器错误
   - Token传递方式：请求头 Authorization: Bearer xxx
   - 分页参数：page（从1开始）、size（默认20）

3. **分配第一批任务**（30分钟）
   - 明确今天各自要做什么
   - 约定今晚站会时间
   - 建立共享文档记录接口定义

4. **环境检查**（15分钟）
   - C确认MySQL已安装并能连接
   - B确认DevEco Studio能正常编译运行
   - A确认Git仓库已创建、分支策略确定

**产出物**：《接口规范文档v0.1》、环境就绪确认、今日任务清单

#### 🌞 下午（14:00-18:00）：分头干活

**后端C的任务（4小时）**：

1. **数据库初始化**（1小时）
   - 导入已有的 medical_health.sql（22张表+1500+条测试数据）
   - 确认所有表都能正常查询

2. **Spring Boot项目配置**（1小时）
   - 配置application.yml（数据库连接、端口8080）
   - 配置MyBatis-Plus、跨域CORS、日志级别

3. **创建通用工具类**（1小时）
   - Result统一返回结果封装
   - JwtUtil工具类（生成/验证Token）
   - BCrypt密码加密工具
   - 全局异常处理器

4. **启动项目并测试**（1小时）
   - 启动Spring Boot应用
   - 用Postman测试Hello World接口
   - 把项目地址告诉前端B（如 http://localhost:8080）

**前端B的任务（4小时）**：

1. **完善HttpUtil.ets网络请求模块**（2小时）
   - 实现get()和post()方法
   - 配置后端地址
   - 实现Token自动附加到请求头
   - 实现统一的错误处理

2. **创建AuthManager.ets认证管理器**（1小时）
   - Token的保存/读取/删除
   - 登录状态检查
   - Token过期判断

## ⚠️ 六、协作风险预案

**风险1：接口格式不一致**（概率60%）⚠️ 最高发
- Day 1就必须定好接口规范文档
- 每次新接口写完后立即更新共享文档
- B发现字段不对立刻找C不要自己猜
- Day 3/7/14三次接口评审会提前暴露问题

**风险2：一方延期导致另一方等待**（概率50%）
- 如果C接口延迟：B先用Mock数据开发UI（但要约定Mock格式）
- 如果B页面延迟：C可以先Postman自测不影响进度
- 每晚站会及时发现延期风险
- 延期超过半天A要介入协调调整计划

**风险3：联调发现大量问题**（概率55%）
- 安排两次联调（中期Day10+全面Day18-19）分散风险
- 问题按P0/P1/P2分级处理
- P0当场修P1尽量修P2记录
- 预留Day 20全天专门修Bug

**风险4：DeepSeek API不稳定**（概率40%）
- 前端做好loading和超时处理
- 后端缓存常见回答
- 准备本地预设答案降级
- 如果完全不可用就用规则引擎生成简单建议

**风险5：沟通不及时导致返工**（概率45%）
- 每天站会雷打不动（除非极特殊情况）
- 共享文档实时更新（接口规范/问题清单/TODO）
- 重要决定必须有会议纪要
- 微信群随时@对方不要憋着不说

**风险6：演示前发现致命Bug**（概率30%）
- Day 21上午充分测试预留缓冲时间
- 准备备用演示方案（视频/GIF离线版本）
- A准备应急答辩话术
- 核心流程至少有2种展示方式

## 💡 八、协作最佳实践

### 前后端配合黄金法则

1. **接口先行**：C写接口前先和B对齐需求，写完后立即通知B
2. **Mock约定**：如果接口没好，B用Mock开发时必须和C约定好数据格式，避免后期大量改动
3. **问题不过夜**：当天发现的接口问题当天解决，不要积压
4. **文档同步**：共享文档（接口规范/问题清单）必须实时更新，不要只在脑子里
5. **互相理解**：C要理解B的UI需求可能变化，B要理解C的数据结构可能有约束
6. **站会不流于形式**：真正汇报进度和阻塞，不是走过场

### 联调技巧

1. **B操作C看日志**：B在前端点，C在后端看日志，效率最高
2. **A专职记录**：A不写代码只记录问题，避免遗漏
3. **小问题当场修**：10分钟能修好的不要记录到明天
4. **大问题定位原因**：修不好的先定位根因，约定修复时间
5. **截图留存证据**：每个Bug都截图，方便后续追踪

### 冲突解决机制

1. **技术分歧**：A做最终决定（基于技术最优原则）
2. **进度分歧**：以核心流程优先为原则砍掉非必要功能
3. **质量分歧**：以演示能用为最低标准，完美是加分项
4. **沟通不畅**：立即面对面开会（不要在微信上扯皮）
