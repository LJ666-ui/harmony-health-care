### ⚠️ V7.1 关键变更说明

```
❌ V7.0 设计:
   前端浏览 → POST /browse/record → 后端 user_browse_history 表 → 推荐算法

✅ V7.1 改进:
   前端浏览 → BrowseTracker自动记录 → 本地 local_browse_history 表(SQLite)
            → 推荐(DailyRecommendPage直接查本地SQLite)
            → 无需网络! 离线也能用!

变更原因:
1. 减少后端工作量(少建1张表+少写6个API)
2. 提升用户体验(即时记录,无延迟)
3. 支持离线模式(断网也不丢浏览数据)
4. 隐私友好(浏览数据不离开用户设备)
```

#### 🆕 模块C: 浏览记录（纯前端！V7.1核心变更）

```
⚠️ 此模块完全不依赖后端!

用户打开任意页面 (如 HerbalDetailPage)
     │
     ▼ aboutToAppear() 触发
┌──────────────────────┐
│ BrowseTracker.recordStart() │
│ → INSERT INTO local_browse_history │
│   (type='HERBAL', id:1, title:'人参', startTime:now)
└──────────────────────┘
     │
     │ 用户浏览中...
     │
     ▼ aboutToDisappear() 触发
┌──────────────────────┐
│ BrowseTracker.recordEnd()   │
│ → UPDATE SET duration = now-startTime
└──────────────────────┘

=== 数据消费 ===

BrowseHistoryPage (浏览历史):
  SELECT * FROM local_browse_history ORDER BY create_time DESC
  → 渲染时间线列表 (不需要请求后端!)

DailyRecommendPage (每日推荐):
  SELECT target_id, target_title, COUNT(*) as freq
  FROM local_browse_history WHERE create_time > (now - 30天)
  GROUP BY target_id ORDER BY freq DESC LIMIT 10
  → 取Top 10高频浏览对象作为推荐候选
```

## 🔄 四、标准联调工作流程

### 4.1 单接口联调五步法

```
Step 1: C先用Postman自测 (约15分钟/接口)
  ├── 启动后端: mvn spring-boot:run (端口8080)
  ├── Postman发送请求，检查状态码/JSON/错误场景
  └── 截图保存到共享文档，标记"C自测通过"

Step 2: B对接前端代码 (约20分钟/接口)
  ├── ApiConstants确认URL已定义
  ├── 编写HttpUtil调用 + Model映射
  ├── 页面UI渲染 + Loading/Empty/Error处理
  └── 处理边界情况

Step 3: B+C联合调试 (双方在线，约10分钟/接口)
  ├── B操作App触发请求
  ├── C查看IDE控制台日志
  ├── 对比请求数据 vs 响应 vs 页面渲染
  └── 确认正常/异常/边界全部通过

Step 4: 问题修复 (视复杂度 5-60分钟)
  ├── 字段不匹配 → 谁错谁改
  ├── 类型不对 → 加注解 or 做转换
  ├── 500报错 → C查日志修Bug
  └── 重复Step 3验证

Step 5: 验收签字 (2分钟)
  ├── 截图保存
  ├── 更新Excel状态为"已联调 ✓"
  └── 进入下一个接口
```
