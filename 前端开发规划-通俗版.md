# 一、项目是什么？

## 1.1 一句话介绍

**星云医疗助手** = 一个专门给中老年人用的**健康管理手机App的前端界面**

## 1.2 打个比方

想象一下这个系统就像一个**医院的手机端服务大厅**：

```
┌─────────────────────────────────────────────────────┐
│                    星云医疗助手                        │
│                                                       │
│  前端（你们要做的）          后端（服务器大脑）           │
│  ┌───────────┐         ┌──────────────────┐           │
│  │ 用户看到的  │  ←──→   │ 处理所有业务逻辑    │           │
│  │ 手机屏幕    │         │ 存储所有数据       │           │
│  └───────────┘         └──────────────────┘           │
│         ↑                      ↑                       │
│    老年人操作简单           数据安全可靠                │
│    字体大、按钮大            API接口调用                │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**前端（你们已经做好的基础）**: 
- ✅ 28个页面（登录、首页、医院、预约、健康记录、康复、科普等）
- ✅ 12个通用组件（按钮、卡片、输入框、加载动画等）
- ✅ 使用鸿蒙系统（HarmonyOS/ArkTS）开发
- ✅ 已完成GlobalTheme标准化适配
- ✅ 已实现Mock数据展示

**前端（还需要完善的）**:
- 🔌 对接后端API接口（替换Mock数据）
- 🎨 完善交互体验（动画、反馈、提示）
- 🔒 实现用户认证流程（登录态管理）
- 📱 优化适老化设计（字体、间距、操作简化）
- 🧪 完善表单验证和错误处理
- 🔄 实现数据本地缓存和离线支持

# 二、前端要做什么？（8大功能模块）

根据现有的28个页面，我们把前端功能分成**8个独立模块**：

## 📦 模块1：用户中心（相当于"个人档案室"）

**作用**: 管理用户的个人信息、登录状态、隐私设置

### 包含的页面：
| 页面 | 文件名 | 功能描述 |
|------|--------|---------|
| ✅ 登录页 | `Login.ets` | 手机号+密码登录，记住密码 |
| ✅ 启动页 | `Index.ets` | 品牌展示页，自动跳转首页或登录页 |
| ✅ 个人中心 | `Profile.ets` | 展示头像、昵称、手机号，进入设置 |
| ✅ 隐私设置 | `PrivacySettingPage.ets` | 隐私开关、数据授权管理 |
| ✅ 隐私政策 | `PrivacyPolicyPage.ets` | 展示隐私政策内容 |

### 要实现的功能：
| 功能 | 通俗解释 | 技术要点 |
|------|---------|---------|
| ✅ 用户登录 | 输入手机号密码，调后端接口验证 | 表单验证 + HTTP请求 + Token存储 |
| ✅ 登录态保持 | 关闭App再打开不用重新登录 | AppStorage持久化Token |
| ✅ 退出登录 | 清除登录状态，回到登录页 | 清除Token + 路由跳转 |
| ✅ 头像上传 | 选择图片上传到服务器 | 图片选择 + 文件上传 |
| ✅ 个人信息编辑 | 修改昵称、性别、年龄等 | 表单编辑 + 数据提交 |
| ✅ 隐私设置 | 开关控制数据收集权限 | Switch组件 + 状态同步 |

**打个比方**: 就像你的个人档案室，存放你的身份证照片、联系方式，还能决定谁可以看你的信息。

# 三、前端长什么样？（项目结构图解）

## 3.1 整体架构（大白话版）

```
┌─────────────────────────────────────────────────────────────┐
│                   前端应用（HarmonyOS App）                  │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐     │
│  │              页面层（Pages）- 28个页面               │     │
│  │  相当于"各个科室的窗口"，用户看到的具体界面           │     │
│  │  例如：HomePage / Login / HospitalPage ...          │     │
│  └──────────────────────┬──────────────────────────────┘     │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────┐     │
│  │              组件层（Components）- 12个组件          │     │
│  │  相当于"通用零件"，多个页面共用的UI元素              │     │
│  │  例如：Button / Card / Input / Loading ...          │     │
│  └──────────────────────┬──────────────────────────────┘     │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────┐     │
│  │              工具层（Utils & Services）              │     │
│  │  相当于"工具箱"，提供各种辅助功能                    │     │
│  │  例如：HttpUtil / RouterUtil / HealthUtils ...      │     │
│  └──────────────────────┬──────────────────────────────┘     │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────┐     │
│  │              状态管理层（Store）                      │     │
│  │  相当于"中央仓库"，统一管理全局数据                 │     │
│  │  例如：AppStore（用户信息、主题配置）                │     │
│  └──────────────────────┬──────────────────────────────┘     │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────┐     │
│  │              样式层（Styles & Theme）                │     │
│  │  相当于"装修风格"，统一的视觉规范                    │     │
│  │  例如：GlobalTheme / globalStyles / style.ets       │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐     │
│  │              后端API接口                             │     │
│  │  通过HTTP请求获取真实数据                            │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

```
✅ 每日同步：每天下班前合并代码，解决冲突
✅ 文档先行：修改公共组件前先沟通，更新README
```

# 八、紧急情况怎么办？

## 8.1 常见问题及解决方案

### ❌ 问题1：编译报错

**现象**: `Error: XXXX` 

**解决方案**:
1. 先看错误信息，定位到哪个文件的哪一行
2. 复制错误信息到百度/Google搜索
3. 检查是否缺少导入（import）
4. 检查类型是否匹配
5. 检查组件属性是否拼写正确
6. 如果还是不行，截图发群里问

**预防措施**:
- 写完一个功能就编译一次，不要攒一堆最后编译
- 使用IDE的语法检查，红波浪线及时修

---

### ❌ 问题2：API对接不通

**现象**: 调用接口返回错误或超时

**解决方案**:
1. 检查后端服务是否启动（问后端同事）
2. 检查API地址是否正确（ApiConstants.ets）
3. 检查请求参数是否齐全
4. 检查Token是否有效（可能过期了）
5. 用Postman手动测试接口是否通
6. 查看控制台Network面板，看请求和响应

**临时方案**:
- 后端没做好之前，继续用Mock数据
- 定义好接口格式，后续替换即可

---

### ❌ 问题3：代码合并冲突

**现象**: Git pull时出现冲突

**解决方案**:
1. 不要慌！冲突是正常的
2. 看清楚冲突的部分（<<<<<<< 和 ======= 之间）
3. 和对方沟通，确定保留谁的代码
4. 手动编辑，删掉不需要的代码和冲突标记
5. 重新add + commit + push

**预防措施**:
- 每天下班前push一次
- 不要同时改同一个文件
- 改公共组件前先说一声

---

### ❌ 问题4：某个功能做不完

**现象**: 时间不够，预估有误

**解决方案**:
1. **第一时间说！** 不要等到deadline才说
2. 说明还剩什么，预计还需多久
3. 和组长商量：
   - 是否可以降低优先级（先做MVP版本）
   - 是否可以找人帮忙
   - 是否可以延后到下一迭代
4. 调整后的计划要及时同步给所有人

**切记**: 早说比晚说好100倍！

---

### ❌ 问题5：需求不清楚

**现象**: 不知道某个功能具体怎么做

**解决方案**:
1. 先看后端规划文档，了解这个功能的目的是什么
2. 参考同类App的做法（平安好医生/丁香医生/微医）
3. 问产品经理或需求方
4. 问后端同学，了解数据结构
5. 先做个简单版本，再逐步完善
