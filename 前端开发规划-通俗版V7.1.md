## 🗄️ 二、SQLite本地数据库设计（前端B独家负责）

### 2.1 数据库概述

> **技术选型**: HarmonyOS RelationalStore (@ohos.data.relationalStore)
>
> **数据库位置**: `entry/src/main/ets/database/local.db`
>
> **创建时机**: App首次启动时在EntryAbility.ets中自动初始化
>
> **重要变更(V7.1)**: 浏览记录完全存储在前端SQLite，不再需要后端的user_browse_history表！
>
> **设计原则**: 离线优先(Offline-First) — 即使没网，核心数据也能用

### 2.2 表结构设计（8张表）

```sql
-- ============================================
-- 前端本地SQLite数据库 V7.1
-- 创建位置: entry/src/main/ets/database/
-- 初始化时机: EntryAbility.onCreate() 时调用 RdbHelper.init()
-- ============================================

-- 【表1】local_user_cache - 用户登录状态缓存
-- 用途: 记录登录Token和基本信息，实现"记住我"/自动登录
-- 数据来源: 用户登录成功后写入 / 从后端/user/info获取
CREATE TABLE IF NOT EXISTS local_user_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,              -- 用户ID(来自后端)
    token TEXT NOT NULL,                   -- JWT令牌(来自后端/login)
    nickname TEXT,                         -- 昵称
    avatar_url TEXT,                       -- 头像URL
    role_type TEXT,                        -- 角色类型
    last_sync_time INTEGER,                -- 最后一次与服务器同步的时间戳(ms)
    expire_time INTEGER                    -- Token过期时间戳(ms)
);
-- 预期数据量: 永远只有1条记录(当前登录用户)

-- 【表2】offline_health_record - 离线健康记录队列
-- 用途: 用户在无网络时录入的健康数据(血压/血糖/心率)，暂存本地
-- 数据来源: 用户手动输入(HealthPage)
-- 同步策略: 恢复网络后批量POST到后端/health-record/add
CREATE TABLE IF NOT EXISTS offline_health_record (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    record_type TEXT NOT NULL,             -- 'blood_pressure'/'blood_sugar'/'heart_rate'/'weight'
    value REAL NOT NULL,                   -- 数值
    unit TEXT,                             -- 单位 'mmHg'/'mmol/L'/'bpm'
    record_time INTEGER NOT NULL,          -- 测量时间戳(ms)
    notes TEXT,                            -- 备注
    synced INTEGER DEFAULT 0,              -- 0=未同步 1=已同步成功 2=同步失败需重试
    create_time INTEGER,                   -- 本地创建时间戳(ms)
    retry_count INTEGER DEFAULT 0          -- 重试次数(超过3次标记为失败)
);
-- 预期数据量: 每用户约10-50条(定期清理已同步的旧数据)

-- 【表3】local_article_cache - 文章阅读缓存
-- 用途: 已读过的文章摘要缓存，支持离线查看标题和摘要
-- 数据来源: 用户打开文章详情时从后端/article/{id}获取并缓存
CREATE TABLE IF NOT EXISTS local_article_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    article_id INTEGER UNIQUE NOT NULL,    -- 对应后端health_article.id
    title TEXT,                            -- 文章标题
    content_summary TEXT,                  -- 内容摘要(截取前200字节省空间)
    cover_image TEXT,                      -- 封面图URL
    category_id INTEGER,                   -- 分类ID
    is_collected INTEGER DEFAULT 0,        -- 是否已收藏(冗余字段加速查询)
    cached_time INTEGER,                   -- 缓存时间戳(ms)
    last_read_time INTEGER                 -- 最后阅读时间戳(ms)
);
-- 预期数据量: 每用户约50-200条(最近阅读的文章)

-- 【表4】local_device_cache - 鸿蒙设备配对缓存
-- 用途: 缓存已经配对过的鸿蒙设备信息，避免重复配对
-- 数据来源: 用户通过DeviceDiscoveryPage配对成功后写入
CREATE TABLE IF NOT EXISTS local_device_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    device_id TEXT UNIQUE NOT NULL,        -- 设备唯一标识(鸿蒙deviceId)
    device_name TEXT,                      -- 设备显示名称
    device_type TEXT,                      -- 设备类型 phone/tablet/watch/large_screen
    device_model TEXT,                     -- 设备型号
    paired INTEGER DEFAULT 0,              -- 是否已配对(0=仅发现 1=已配对)
    last_connect_time INTEGER              -- 最后连接时间戳(ms)
);
-- 预期数据量: 每用户约1-5台设备

-- 【表5】sync_queue - 数据同步任务队列
-- 用途: 记录所有需要同步到服务器的操作(INSERT/UPDATE/DELETE)
-- 数据来源: 离线操作时自动入队
-- 消费方式: SyncManager后台定时检查，有网时逐条执行
CREATE TABLE IF NOT EXISTS sync_queue (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    table_name TEXT NOT NULL,              -- 要同步的本地表名
    operation_type TEXT NOT NULL,          -- INSERT / UPDATE / DELETE
    data_json TEXT NOT NULL,               -- 操作数据的完整JSON
    target_url TEXT,                       -- 对应的后端API URL
    retry_count INTEGER DEFAULT 0,         -- 已重试次数
    status TEXT DEFAULT 'pending',         -- pending / syncing / synced / failed
    error_message TEXT,                    -- 失败时的错误信息
    create_time INTEGER,                   -- 入队时间戳(ms)
    sync_time INTEGER                      -- 成功同步的时间戳(ms)
);
-- 预期数据量: 动态变化，定期清理已synced的记录

-- ============================================================
-- 🆕 V7.1 新增: 以下3张表是本次版本的核心新增!
-- ============================================================

-- 【🆕表6】local_browse_history - 本地浏览历史记录 ⭐⭐ 核心新表!
-- 用途: 记录用户在App内的所有浏览行为(即使没有网络也记录!)
-- 数据来源: BrowseTracker.ets 工具类在每个页面onShow/onHide时自动记录
-- 重要说明: 此表完全替代了原计划中的后端 user_browse_history 表!
--           所有浏览数据分析(统计/推荐)都基于此表的本地数据
CREATE TABLE IF NOT EXISTS local_browse_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    browse_type TEXT NOT NULL,             -- 枚举值:
                                          --   ARTICLE      = 科普文章
                                          --   HERBAL       = 药材百科
                                          --   HOSPITAL     = 医院
                                          --   FOOD         = 食疗食谱
                                          --   KNOWLEDGE    = 知识图谱节点
                                          --   REHAB        = 康复训练
                                          --   AI_CHAT      = AI对话
                                          --   ANCIENT_IMAGE= 古医图
    target_id INTEGER NOT NULL,            -- 浏览对象的ID(对应各业务表主键)
    target_title TEXT,                     -- 对象标题快照(避免联查)
    browse_duration INTEGER DEFAULT 0,     -- 停留时长(秒)，0表示未知
    source_page TEXT,                      -- 来源页面路由名(如'pages/HerbalListPage')
    create_time INTEGER                    -- 浏览开始时间戳(ms)
);
-- 预期数据量: 重度使用用户每月约500-2000条
-- 清理策略: 只保留最近30天的数据，超期自动DELETE

-- 【🆕表7】local_search_history - 本地搜索历史
-- 用途: 搜索框输入时的下拉提示(显示最近搜过的关键词)
-- 数据来源: 用户在搜索框输入并提交时自动记录
CREATE TABLE IF NOT EXISTS local_search_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    keyword TEXT NOT NULL,                 -- 搜索关键词原文
    search_count INTEGER DEFAULT 1,        -- 搜索次数(相同关键词累加，用于排序)
    last_search_time INTEGER,              -- 最后搜索时间戳(ms)
    search_scope TEXT DEFAULT 'GLOBAL'     -- 搜索范围 GLOBAL/HERBAL/ARTICLE/FOOD...
);
-- 预期数据量: 每用户约50-100条(去重后的唯一词)

-- 【🆕表8】local_collection_cache - 本地收藏夹缓存
-- 用途: 收藏的内容(文章/药材/食疗)离线也能查看标题和入口
-- 数据来源: 用户点击收藏按钮时同时写本地+调后端API(双写)
-- 一致性保障: 以服务器为准;本地作为离线兜底
CREATE TABLE IF NOT EXISTS local_collection_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    collect_type TEXT NOT NULL,            -- ARTICLE / HERBAL / FOOD / PRESCRIPTION / KNOWLEDGE
    target_id INTEGER NOT NULL,            -- 收藏对象的主键ID
    target_title TEXT,                     -- 对象标题快照
    target_image TEXT,                     -- 对象封面图URL(可选)
    collect_time INTEGER,                  -- 收藏操作时间戳(ms)
    note TEXT,                             -- 用户备注("这个对我有用")
    is_synced INTEGER DEFAULT 0            -- 0=仅本地 1=已同步到服务器
);
-- 预期数据量: 每用户约20-100条
```

### 2.3 各表详细数据流转说明

#### 📊 表6 local_browse_history 数据生命周期

```
用户打开任意页面
       │
       ▼
BrowseTracker.recordStart() 自动调用  ← 在每个页面的 aboutToAppear() 中触发
       │
       ├─ 写入 local_browse_history 表 (browse_duration=0, create_time=now)
       │
       ▼
用户在该页面浏览... (可能滑动/点击/阅读)
       │
       ▼
用户离开页面 (返回/跳转)
       │
       ▼
BrowseTracker.recordEnd() 自动调用  ← 在每个页面的 aboutToDisappear() 中触发
       │
       ├─ 更新该条记录: browse_duration = 当前时间 - create_time
       └─ 如果时长<2秒，视为"误触"，可选择性删除该条记录
       
=== 数据消费场景 ===

场景A: BrowseHistoryPage 展示浏览历史
  SELECT * FROM local_browse_history 
  ORDER BY create_time DESC 
  LIMIT 50;
  → 渲染为时间线列表

场景B: 统计饼图 (按类型分布)
  SELECT browse_type, COUNT(*) as cnt 
  FROM local_browse_history 
  WHERE create_time > (当前时间 - 7天*24*3600*1000)
  GROUP BY browse_type;
  → 传给前端图表组件渲染

场景C: 个性化推荐 (DailyRecommendPage)
  SELECT target_id, target_title, browse_type, COUNT(*) as freq
  FROM local_browse_history
  WHERE create_time > (当前时间 - 30天*24*3600*1000)
  GROUP BY target_id, target_title, browse_type
  ORDER BY freq DESC
  LIMIT 20;
  → 取Top 10高频浏览的对象，作为推荐候选
  → 结合后端/herbal/recommend 或 /article/recommend 进一步筛选
```

#### 📊 表3 local_article_cache 数据流转

```
用户点击文章进入 ScienceDetailPage
       │
       ▼
aboutToAppear() 中:
  ① 先查本地: SELECT * FROM local_article_cache WHERE article_id = ?
  ② 如果命中 → 直接渲染标题和摘要(快速显示)
  ③ 同时异步请求后端: GET /article/{id} 获取完整内容
  ④ 收到响应后 → 更新缓存 + 渲染完整内容
  ⑤ 如果未命中 → 显示Loading → 请求后端 → 写入缓存 → 渲染

缓存失效策略:
  - 手动清理: 设置页提供"清除缓存"按钮
  - 自动过期: cached_time 超过7天的记录下次查询时删除
  - 存储上限: 超过200条时删除最旧的
```

## 🎨 六、新增页面详细设计要点

### 6.1 药材列表页 (HerbalListPage.ets) — 核心交互流程

```
页面加载
  ↓
① 调用 GET /herbal/list?page=1&size=20&category=全部
  ↓ (如果USE_MOCK=true则读herbalMock.ts)
② 渲染 CategoryTabs: [全部][补气][补血][清热][解表][...]
  ↓
用户操作分支:
  ├─ 点击分类Tab → 重新请求 ?category=xxx
  ├─ 点击筛选栏 → 展开/折叠药性/味/排序选项
  ├─ 输入搜索框 → 防抖300ms → GET /herbal/search?keyword=xxx
  ├─ 下拉刷新 → page=1 重新加载
  └─ 滚动到底部 → page++ 加载更多(无限滚动)
  ↓
③ 渲染 WaterfallGrid (2列):
  ┌──────────┐ ┌──────────┐
  │ [药材图片]│ │ [药材图片]│
  │ 人参      │ │ 当归      │
  │ 补气·微温 │ │ 补血·温   │
  │ 大补元气..│ │ 补血活血..│
  └──────────┘ └──────────┘
  ↓
④ 点击某卡片 → RouterUtil.push('pages/HerbalDetailPage', {id: item.id})
  → 同时 BrowseTracker.record('HERBAL', id, name)  ← 自动记录浏览!
```

### 6.2 知识图谱页 (KnowledgeGraphPage.ets) — Canvas渲染流程

```
页面加载
  ↓
① 调用 GET /knowledge/graph?centerId=1&depth=2&maxNodes=50
  ↓ 返回 { nodes: [{id,x,y,name,nodeType,...}], edges: [{sourceNodeId,targetNodeId,...}] }
② GraphCanvas.render(nodes, edges)
  ↓
  ├─ 清空画布 context.clearRect(0,0,width,height)
  ├─ 遍历edges数组 → drawLine(source.x,source.y,target.x,target.y,colorByType)
  └─ 遍历nodes数组 → drawCircle(x,y,radius,colorByNodeType) + fillText(name)
  ↓
③ 用户触摸事件处理:
  ├─ TouchType.Down → checkNodeTap(x,y) → 选中节点 → 高亮描边
  ├─ TouchType.Move → 判断单指/双指 → 平移/缩放
  └─ TouchType.Up → 弹出NodeDetailPanel(显示节点详情)
  ↓
④ 双击节点 → 以该节点为新中心重新请求 graph?centerId=newId
⑤ 类型筛选Tab → 前端过滤 nodes.filter(n => n.nodeType === selectedType)
```

### 6.3 浏览历史页 (BrowseHistoryPage.ets) — 纯本地数据!

```
页面加载
  ↓
① 查询本地SQLite (不需要网络!):
   SELECT * FROM local_browse_history ORDER BY create_time DESC LIMIT 100
  ↓
② 按日期分组渲染时间线:
   今天
   ├─ 14:30 📖 《高血压防治指南》 停留2分钟
   ├─ 13:15 💊 人参 - 药材详情       停留1分钟
   └─ 10:00 🧠 知识图谱-高血压       停留3分钟
   
   昨天
   ├─ 18:20 🍜 红枣银耳汤做法       停留1分钟
   └─ 09:30 🏥 XX医院内科           停留30秒
  ↓
③ 顶部统计饼图:
   SELECT browse_type, COUNT(*) FROM local_browse_history
   WHERE create_time > (now - 7天)
   GROUP BY browse_type
   → 传给StatPieChart组件渲染环形图
  ↓
④ 右上角"清空"按钮 → DELETE FROM local_browse_history → 确认提示
```
